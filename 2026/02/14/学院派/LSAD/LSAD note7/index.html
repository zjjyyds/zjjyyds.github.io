1. 为什么要学安全（Why do we care?）

1.1 现实世界的“安全”不是玄学，是事故

PPT举了几个典型场景：医院勒索软件、金融机构泄露。

  • 医院被勒索软件攻击

    • 勒索软件 = 攻击者加密你的关键数据/系统，然后要赎金才给解密

    • 影响不是“钱”，而是：

      • 挂号系统瘫痪

      • 检验结果系统不可用

      • 医疗设备依赖网络系统

    • 本质是打击 Availability（可用性） + Integrity（完整性）（有时还会偷数据打击机密性）

  • 金融机构/征信机构泄露（Equifax / First American Financial）

    • 典型泄露内容：身份证号、地址、工作信息、信用记录

    • 结果：

      • 直接财产损失（诈骗/盗刷）

      • 长期身份风险（你无法“换一个身份证号”）

    • 本质是 Confidentiality（机密性）破坏

1.2 对个人：隐私 + 职业稳定

  • 你自己作为用户：账号密码、照片、通讯录、定位信息、支付记录

  • 你作为开发者/运维：你写的系统一旦出事故，你可能要背锅（“Job security!”）

1.3 对行业：安全岗位增长快

PPT提到 2022–2032 预测 33% 增长。
你可以理解成：

  • 系统越来越复杂、联网越来越广

  • 攻击面越来越大

  • 安全是“必须投入的成本”

2. 安全基本原则（Basic Principles）——每一条都讲透

PPT列了五条，并强调：最重要：Threat Model（威胁模型）

2.1 Security is economics（安全是经济学 / 资源分配）

核心意思：安全不是“开关”，是“投入-收益”问题。

  • 你永远不可能做到“绝对安全”

  • 安全措施会消耗：

    • 开发成本（加密、审计、权限系统）

    • 性能成本（加密/校验/多因子）

    • 用户体验成本（更难登录、更麻烦操作）

    • 运维成本（监控、备份、演练）

所以你要做的是：

  • 明确最重要的资产

  • 把资源投入到“风险最大/损失最大”的地方

例子：

  • 个人博客：可能不需要上硬件安全模块（HSM）

  • 银行系统：必须投入极高成本（审计、合规、密钥管理、双人审批、隔离环境）

2.2 Least privilege（最小权限原则）

定义：只给完成任务所必须的最少权限。

为什么重要？

  • 权限越大，误操作/被盗用的后果越大

  • 攻击者拿到一个账号后，能做的事取决于权限范围
    => 权限越小，攻击者收益越小

例子1：Linux 文件

  • 你把私钥 ~/.ssh/id_rsa 设成 chmod 644（别人可读）

    • 同机器上其他用户直接读走你的私钥

    • 等于“把你家钥匙复印件贴在门上”

例子2：数据库账号

  • Web 服务只需要 SELECT 某些表

  • 你却给了 DROP TABLE 权限
    => SQL 注入直接“删库跑路”

例子3：云权限

  • 某个 CI/CD 只需要读镜像仓库

  • 给了管理员权限
    => 一旦 token 泄露，云资源全被挖矿

2.3 Defense in depth（纵深防御：层层防护）

定义：不要依赖单一防线；即便一层破了，还有下一层。

为什么必要？

  • 单点防御的假设是“这层永远不会破”

  • 现实是：总会破（0day、配置错误、人为失误）

常见分层：

  • 网络层：防火墙/WAF、分段

  • 应用层：鉴权、权限控制、输入校验

  • 数据层：加密、最小权限

  • 运维层：监控告警、审计日志

  • 组织层：制度、培训、应急响应

例子：

  • 你的网站防 SQL 注入（输入校验）是第一层

  • 即使 SQL 注入成功，数据库账号也应该是只读（第二层）

  • 即使数据库读到了数据，敏感字段应该加密（第三层）

  • 即使密文泄露，密钥应在 KMS/HSM，不在代码里（第四层）

2.4 Complete mediation（完全仲裁/持续认证）

PPT解释：continuous authentication（持续认证）。

定义：每次访问资源都要检查权限，而不是“登录一次就永远放行”。

为什么重要？

  • 权限可能变化（员工离职、角色调整）

  • 会话可能被劫持（cookie 被偷）

  • 攻击可能发生在登录之后（内部横向移动）

例子：

  • 你登录公司系统后，半小时后被 HR 取消权限

  • 如果系统只在登录时检查，你仍能继续操作
    → 完全仲裁要求：每次关键请求都要做权限判断

现实实现方式：

  • API 每次请求携带 token（JWT/Session）

  • 服务端每次验证 token + 查权限

  • 对高风险操作要求重新验证（输密码/短信/硬件 key）

2.5 Accounting for human factors（考虑人为因素）

PPT说：errors（错误）。

核心：人会犯错，系统必须设计得“犯错也不至于爆炸”。

常见人为错误：

  • 把密钥提交到 GitHub

  • 把对象存储桶设成 public

  • 把权限 chmod 成 777

  • 把生产数据库当测试库删数据

  • 点击钓鱼邮件

安全设计如何“容错”：

  • 默认安全（secure by default）

  • 自动扫描泄露（secret scanning）

  • 权限最小化 + 逐级审批

  • 高风险操作二次确认

  • 审计日志 + 告警

⭐ 2.6 最重要：Threat Model（威胁模型）

PPT强调：Most important: know your threat model。

威胁模型要回答 4 个问题（非常常用）：

  1. 资产（Assets）：你要保护什么？

    • 个人：照片、聊天记录、密码、银行卡

    • 公司：用户数据、源代码、密钥、资金、可用性

  2. 攻击者（Adversary）：谁会攻击你？

    • 路人黑客、竞争对手、有组织犯罪、国家级

  3. 能力（Capabilities）：攻击者能做什么？

    • 能否物理接触机器？

    • 是否能监听网络？

    • 是否能社工/钓鱼？

    • 是否能买 0day？

  4. 后果（Impact）：被攻破损失多大？

    • 金钱、隐私、信誉、法律责任、生命安全

为什么它是最重要？
因为所有安全措施都是“对威胁的回应”，没有威胁模型就会出现：

  • 过度安全（浪费钱/性能/体验）

  • 安全错位（防了不重要的，重要的没防）

PPT列：Confidentiality / Integrity / Authenticity / Availability。

我按“定义 → 典型攻击 → 常用手段 → 例子”给你展开。

4. Confidentiality（机密性）——加密体系讲透

4.1 基本概念：明文、密文、密钥

PPT定义：plaintext、ciphertext、key。

  • 明文 Plaintext：你要保护的原始数据

  • 密文 Ciphertext：看起来像乱码的结果（攻击者看到的）

  • 密钥 Key：加密/解密必须知道的秘密

加密的目标不是“让它看起来复杂”，而是：没有密钥就无法恢复明文。

4.2 对称加密（Symmetric Crypto）

PPT：同一个 key 用来加密和解密。

优点：

  • 快（适合大量数据：硬盘加密、传文件）

  • 实现成熟（AES、ChaCha20）

缺点：

  • 密钥怎么给对方？（密钥分发问题）

    • 如果你用明文把 key 发过去，攻击者截获就完了

典型场景：

  • 磁盘加密（你输入密码→派生密钥→解密硬盘）

  • VPN 传输数据最终也是对称加密

4.3 非对称加密（Asymmetric / Public Key）

PPT：公钥加密，私钥解密；公钥可分发，私钥必须保密。

优点：

  • 解决“密钥分发”问题

    • 你把公钥公开，任何人都可以给你发加密信息

    • 只有你有私钥能解

缺点：

  • 慢（不适合直接加密大文件）

经典用法：

  • 混合加密（现实 HTTPS 就是这么干）

    1. 用非对称加密协商一个临时对称密钥（session key）

    2. 大量数据用对称加密传输（快）

4.4 PPT提到 age demo（你可以当作“加密工具实例”）

PPT有 “Quick demo with age”。
你不需要记工具细节，但要理解：

  • age 是现代文件加密工具

  • 支持公钥加密、对称加密

  • 用它演示“密文不可读、密钥可恢复”

5. Integrity（完整性）——哈希 / 校验 / 密码存储

5.1 哈希函数是什么

PPT：任意长度数据 → 固定长度 bit 字符串（hash）；像“摘要”。

核心直觉：

  • 哈希像“指纹”

  • 你改了一个字符，指纹都变了

5.2 密码学哈希的关键性质（你做题常考）

PPT说：“很难找到两个输入有相同 hash”。
再加上后面“很难从 hash 反推输入”。

整理成三条（考试常见）：

  1. 抗碰撞（Collision resistance）
    找到 x != y 但 hash(x)=hash(y) 很难

  2. 抗二次原像（Second preimage resistance）
    给定 x，很难找 y≠x 使得 hash(y)=hash(x)

  3. 单向性（Preimage resistance）
    给定 h，很难找 x 使得 hash(x)=h（不可逆）

5.3 Hash-based MAC（消息认证码）与校验下载

PPT：

  • MAC：给消息打 hash tag，收件人重新 hash 比对即可发现篡改

  • Checksums：下载文件后算 hash 与官方提供的 hash 比，不必逐位比对

你要记住：

  • hash 可以检测篡改

  • 但 hash 本身不证明“是谁发的”（认证不够）

5.4 密码存储为什么存 hash 不存明文

PPT：服务器被攻破时，泄露的是 hash；密码无法从 hash 恢复。

这里要补一个现实关键点（很重要）：

  • 真实系统不会直接 hash(password)

  • 会加 salt，并使用慢哈希（bcrypt/scrypt/Argon2）

    • 防彩虹表

    • 防 GPU 暴力破解

（PPT没写 salt，但你写报告/答题提一句会更专业。）

5.5 sha256sum demo 的意义

PPT：用 sha256sum 验证下载的 ISO 等。
你理解成：

  • 你下载 Ubuntu ISO

  • 官网给你一个 sha256 值

  • 你本地算出来对比

  • 一样 => 没被镜像站/中间人篡改

6. Authentication / Authenticity（认证/真实性）——数字签名

PPT标题写 Authentication，目标写 Authenticity（本质就是“证明来源”）。

6.1 签名是什么

PPT：用私钥签名，公钥验证。

  • 签名生成： file + private_key -> signature

  • 签名验证： signature + public_key -> verify

为什么能证明来源？

  • 私钥必须保密

  • 只有拥有私钥的人才能生成能被公钥验证通过的签名
    ⇒ 验证通过 = “确实是私钥持有人签的”

6.2 签名还能顺便保证完整性

因为签名通常是对文件的 hash 做签名：

  • 文件变了 → hash 变了 → 签名验证失败

所以签名 = 来源证明 + 防篡改

6.3 ssh demo 的意义

PPT：Demo with ssh。
SSH 登录时：

  • 你用私钥证明“你是你”

  • 服务器用公钥验证

  • 同时服务器也会让你验证“服务器真的是那台服务器”（known_hosts）

7. Availability（可用性）——运维安全非常常考

PPT列：Filtering、Load balancers、Redundancies、Backups。

我给你展开成“攻击 → 对策”。

7.1 常见打可用性的攻击

  • DDoS：海量请求把服务打崩

  • 勒索软件：数据被加密导致业务不可用

  • 单点故障：某个组件挂了全挂（数据库、网关）

  • 人为误操作：删库、误配置、发布失败

7.2 Filtering（过滤）

目标：让恶意流量 到不了核心服务

  • WAF（Web Application Firewall）

  • rate limit（限流）

  • 黑名单/信誉系统

  • 验证码/挑战（防机器人）

7.3 Load Balancer（负载均衡）

目标：把流量分给多台机器

  • 避免单台过载

  • 支持横向扩展（scale out）

7.4 Redundancy（冗余）

目标：某个组件挂了还能顶上

  • 多副本服务

  • 主从数据库

  • 多可用区部署

7.5 Backups（备份）

目标：系统挂了能恢复到最近状态

  • 全量备份 + 增量备份

  • 定期演练恢复（否则备份等于没有）

  • 防勒索：备份要隔离（不可直接被覆盖/加密）

8. File Security：UNIX 权限与所有权（这块你实验/面试超常用）

PPT从“UNIX 多用户”讲到权限模型与 chown/chmod。

8.1 为什么需要权限

  • 多用户机器上，别人也能登录

  • 你有敏感文件（比如私钥）

  • 你希望只有你能读/改它

8.2 三类“所有者”

PPT：user / group / others。

  • owning user：文件属于谁

  • owning group：文件属于哪个组

  • others：除了以上两类的所有人

8.3 三种权限 r/w/x 到底意味着什么（很多人只会背字母）

PPT展示 read/write/execute。

对“普通文件”：

  • r：能读文件内容（cat、less）

  • w：能修改内容（写入/覆盖）

  • x：能执行（把它当程序跑）

对“目录”：

  • r：能列出目录里的文件名（ls）

  • w：能在目录里创建/删除/重命名文件（前提通常还要 x）

  • x：能进入目录（cd），能访问目录中已知名字的文件

目录的 x 超关键：

  • 没有 x，你连 cd 都进不去

  • 有 r 没 x，你能看到名字但进不去/访问不了

8.4 ls -l 那串字符怎么读

PPT提 ls -l to see permissions。

示例：drwxr-xr--

  • 第 1 位：d 目录，- 文件

  • 后 9 位：三组三位

    • user：rwx

    • group：r-x

    • others：r--

8.5 改所有者：chown

PPT给格式：[sudo] chown [-R] newuser:newgroup [FILES]

  • sudo：改所有权一般需要管理员权限

  • -R：递归（目录下所有文件一起改）

  • newuser:newgroup：

    • newuser 变更拥有者

    • newgroup 变更所属组

    • 你也可以只改 user 或只改 group（具体写法不同系统略有差异）

典型场景：

  • 你从 root 创建了一个文件，普通用户读不了
    → sudo chown yourname:yourgroup file

8.6 改权限：chmod

PPT给格式：[sudo] chmod [-R] [permissions] [FILES]

两种常见写法：

写法A：符号法（好理解）

  • chmod o+r file：others 增加读

  • chmod g-w file：group 去掉写

  • chmod u+x script.sh：user 增加可执行

写法B：数字法（常用在运维）

  • chmod 600 private_key

    • user：rw-

    • group：—-

    • others：—-

  • chmod 644 file

    • user：rw-

    • group：r—

    • others：r—

8.7 “为什么重要？”——私钥权限错误的后果

PPT问：如果你把私钥设成这种权限会怎样？

你可以直接写成一句话的结论：

  • 私钥如果对 group/others 可读
    ⇒ 同机其他用户能复制你的私钥
    ⇒ 可以伪装你登录服务器
    ⇒ 相当于“身份被盗用”

所以现实 SSH 会直接拒绝：

  • 私钥权限过宽时，ssh 会提示 “Permissions are too open” 并拒绝使用

9. 签名与证书（Certificates）——解决“公钥到底信不信”的问题

PPT提出关键疑问：
“我给你一个公钥，说它是银行的，你怎么验证？”

9.1 问题本质：公钥容易被冒充

  • 数字签名能证明“签名者拥有某私钥”

  • 但你不知道这个公钥是否真属于银行

  • 攻击者可以：

    • 给你一个“假银行公钥”

    • 你用它加密/验证，就被中间人接管

这就是 信任引导（bootstrapping trust） 的难题。

9.2 证书是什么

PPT：证书 = 被签名的消息，表示“信任某公钥”。

通俗版：

  • 证书就像“身份证/营业执照”

  • 它把：

    • “某个域名/组织”

    • 和 “某个公钥”

    • 绑定在一起

  • 并且由一个你信任的机构签名

9.3 根证书与信任链（Chain of Trust）

PPT：OS 内置 root certificates；证书是链式签名，链到 root 就算可信。

你可以这样理解：

  • 你的电脑/手机系统里，预装了一堆“可信 CA 的公钥”（根证书）

  • 网站证书通常不是根证书直接签的，而是：

    • 网站证书 ← 中间 CA ← 根 CA

  • 浏览器验证：

    1. 网站证书签名是否有效

    2. 证书链是否能一路追溯到某个系统内置根证书

    3. 域名是否匹配

    4. 是否过期/吊销

PPT也提醒：不是完美系统（wosign/diginotar 事件）。
结论：CA 体系靠“信任 + 监管 + 撤销机制”维持，但也会翻车。

10. HTTPS / SSL certificate（这页虽然短，但你要会解释）

PPT一句话：HTTPS：加密 HTTP 请求，公开验证。

你要会完整解释：HTTPS 解决了两个问题

  1. 机密性：内容不被窃听（加密传输）

  2. 真实性：你连的真的是目标网站（证书验证）

（补充你写报告会更强）TLS 握手直觉：

  • 浏览器拿到网站证书 → 用系统根证书验证它可信

  • 验证通过 → 协商出会话密钥（session key）

  • 后续数据用对称加密传输（快）

11. BeyondCorp & Zero Trust（零信任）——现代访问控制理念

PPT要点：传统“内网 + 防火墙”不行了；远程办公、云服务、网络应用。

11.1 传统模型（Perimeter / Intranet）

  • 假设：

    • 公司内网是可信的

    • 外网是不可信的

  • 做法：

    • 内网给很多权限

    • 外网靠 VPN/防火墙挡

问题：

  • 员工远程办公：人不在内网

  • 服务上云：资源不在“公司机房内网”

  • 内网一旦被入侵：横向移动非常容易（攻击者像进了城）

11.2 零信任模型（Zero Trust）

PPT：动态策略、更细粒度控制；根据用户+设备+其他启发式。

核心思想（很关键）：

  • 不要因为“在内网”就信任

  • 每次访问都基于：

    • 用户身份（是谁）

    • 设备状态（是否合规、有无补丁、是否加密）

    • 环境（地点、时间、风险评分）

    • 行为（是否异常）

你可以一句话记：

Never trust, always verify（从不默认信任，永远持续验证）
