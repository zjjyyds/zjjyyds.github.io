📘 Version Control & Backups（Lecture 8）系统笔记（原文内嵌详细补充版）

一、为什么需要版本控制（Version Control）

1️⃣ 核心作用

版本控制（VC / VCS）的本质目标：

  1. 跟踪代码变化

    • 知道「谁」在「什么时候」改了「什么」

      📌补充：

      • “谁”：commit 里有 author/committer（用户名+邮箱）。

      • “什么时候”：commit timestamp。

      • “改了什么”：diff（行级变化）、文件增删改。

      • 应用场景：线上 bug 出现时追责不是目的，真正目的是快速定位“引入 bug 的提交”。

      • 考试爱问：如何查看谁改的？→ git log / git blame <file>。

    • 可以回滚到任意历史版本

      📌补充：

      • “回滚”有两种常见含义：

        1. 回到过去某个版本（checkout/restore）：临时查看旧版本，不一定产生新提交。

        2. 撤销某个提交的影响（revert）：产生一个“反向提交”，更安全（团队常用）。

      • 常用指令：

        • git log --oneline 找到 commit

        • git checkout <hash>（进入“分离 HEAD”，只看不改）

        • git revert <hash>（安全撤销，推荐）

        • git reset --hard <hash>（危险，会改历史，慎用）

      • 应用场景：线上事故快速回退：更推荐 git revert 或用 release/tag 回滚。

  2. 多人协作

    • 多个人同时改代码

      📌补充：

      • 核心矛盾：同一份代码，多个开发者并行修改。

      • Git 的解决方式：每个人本地都有完整仓库，最后通过合并（merge/rebase）把历史汇总。

    • 自动合并不同人的修改

      📌补充：

      • Git 合并的是提交历史（commits），并最终合并到文件层面。

      • 能自动合并的前提：修改发生在不同文件或同文件不同区域。

      • 常见指令：git merge、git pull（默认=fetch+merge）。

    • 冲突可控、可解决

      📌补充：

      • “可控”= Git 不会瞎合并，遇到冲突会停下来让你决定。

      • “可解决”= 冲突标记 + 手动改 + add + commit。

      • 考试爱考：冲突标记长什么样、解决三步是什么（你后面已经写了）。

  3. 安全地开发新功能

    • 在不影响主线（production）的情况下

      📌补充：

      • 主线= main/master 通常对应可部署版本。

      • 你在主线上直接乱改，会让大家拉到不稳定代码。

      • 所以要用分支隔离风险。

    • 创建新分支测试功能

      📌补充：

      • 分支是“指针”，创建成本几乎为 0，所以鼓励大胆试验。

      • 典型流程：git checkout -b feature-x → 开发 → PR → merge 回主线。

      • 考试爱问：为什么 Git 分支轻量？→ 只是指针，不是复制整个目录。

2️⃣ 常见版本控制系统（VCS）

  • Git（主流、事实标准）

    📌补充：

    • Git 是分布式版本控制（DVCS）。

    • “事实标准”指生态最强：GitHub/GitLab/CI 工具链几乎围绕 Git。

  • Mercurial

  • Subversion（SVN）

  • Perforce

  • Bazaar

    📌补充（考试对比角度）：

    • SVN：集中式（中央服务器权威），分支相对笨重；但某些企业老系统仍在用。

    • Perforce：大规模二进制资产/游戏开发常见。

⚠️ 重要区分：

Git ≠ GitHub

  • Git：版本控制工具（本地就能用）

    📌补充：

    • 你不联网也能：commit、branch、diff、log。

    • 本质：.git/ 目录里保存了历史和对象数据库。

  • GitHub：托管 Git 仓库的云平台

    📌补充：

    • GitHub 提供：远程仓库、PR、issue、actions、权限管理等。

    • 远程平台还有 GitLab、Bitbucket。

二、Git 是什么 & 为什么它牛

1️⃣ Git 的背景

  • 2005 年，Linus Torvalds 为 Linux 内核开发

    📌补充：

    • Linux 内核协作规模极大，需要高性能和强一致性方案。

  • 因 BitKeeper 停止免费授权而诞生

  • 最初 2–3 个月就完成第一版

  • 名字来源：Linus 的自嘲（英式俚语里 Git = 讨厌的人）

    📌补充：

    • 考试一般不考八卦，但可能考：Git 设计为适配大规模协作。

2️⃣ Git 的设计哲学

🔧 Plumbing & Porcelain

  • Plumbing（底层原语）

    • 面向脚本

      📌补充：

      • 适合自动化/组合命令，不强调“好用”。

    • 如对象存储、树结构

      📌补充：

      • 例如底层对象：blob/tree/commit/tag（你后面讲了）。

      • 底层命令例子：git cat-file、git hash-object、git update-ref（了解即可）。

  • Porcelain（高层命令）

    • 面向用户

      📌补充：

      • 更符合人类习惯，默认行为更安全。

    • 如 git add / git commit

      📌补充：

      • git add = 选择哪些改动进入下一次快照

      • git commit = 固化快照为一个 commit 节点

👉 我们日常用的都是 porcelain

📌补充：

  • 考试常问：“为什么说 Git 像工具箱？”→ 因为底层可组合。

3️⃣ Git 的三大“杀手级特性”

✅（1）快

  • 几乎所有操作都在本地

    📌补充：

    • git log/diff/status/branch/commit 都是读写本地 .git/ 数据库。

    • 只有 fetch/pull/push/clone 才需要网络。

  • 不依赖网络

  • 离线也能：

    • 看历史

      📌补充：git log --oneline --graph --decorate --all

    • diff

      📌补充：

      • git diff（工作区 vs 暂存区）

      • git diff --staged（暂存区 vs 上次提交）

      • git diff <a> <b>（任意两个版本）

    • checkout

      📌补充：

      • git checkout <branch> 切分支

      • git checkout <hash> 查看旧提交（会进入分离 HEAD）

✅（2）有完整性（Integrity）

  • 所有对象用 SHA-1 哈希

    📌补充：

    • 每个对象内容→算 hash→作为 ID。

    • 任何一位变化，hash 都会完全不同（雪崩效应）。

    • 所以 Git 对象天然防篡改/防损坏（至少能检测）。

  • 任意修改都会被检测

  • 能防止：

    • 传输损坏

    • 文件被偷偷改

📌补充：

  • 常见说法：Git 更像“内容寻址存储（content-addressed storage）”。

✅（3）存的是“快照”，不是“差异”

  • 每次 commit：

    • 保存整个项目状态

      📌补充：

      • 逻辑上是“完整快照”，但物理存储会复用旧对象，所以不一定真的重复占空间。

      • 好处：切换版本快、历史结构清晰。

  • 底层通过对象复用节省空间

    📌补充：

    • 未改的文件对应 blob 会被复用（指针复用）。

    • packfile 压缩也会让历史更省空间（进阶了解）。

三、Git 内部原理（非常重要，考试爱考）

1️⃣ Git 的数据模型

Git 用的是一个：

有向无环图（DAG, Directed Acyclic Graph）

📌补充：

  • “有向”= commit 指向 parent（从新指向旧）

  • “无环”= 不可能出现某个提交把自己当祖先（时间线不闭环）

  • “图”= 支持分叉（branch）与汇合（merge）

  • 每个节点：一次 commit

    📌补充：

    • commit 节点记录：tree + parent(s) + message + author/date 等元数据。

  • 每个 commit 指向：

    • 它的父 commit

      📌补充：

      • 普通提交：1 个 parent

      • merge 提交：2 个 parent（或更多）

  • 没有环（不能回到未来）

2️⃣ Git 的三种核心对象

🧱 Blob（文件）

  • 文件内容

  • 不包含文件名

📌补充：

  • “不含文件名”= blob 只负责内容；名字属于 tree（目录结构）。

  • 这也是为什么 Git 能复用内容相同的文件对象（不同路径也可以指向同 blob）。

🌳 Tree（目录）

  • 目录结构

  • 指向 blob 或其他 tree

📌补充：

  • tree 像“目录清单”：记录（文件名 → blobID）或（子目录名 → treeID）。

  • 所以：文件名/路径信息在 tree 里。

📌 Commit

  • 指向一个 tree（项目快照）

  • 包含：

    • 作者

    • 时间

    • commit message

    • 父 commit

📌补充：

  • commit 的 tree 指针 = “那一刻项目根目录长什么样”。

  • 所以 checkout 某个 commit，本质是把 HEAD 指向它，并把工作区还原为它的 tree。

3️⃣ 分支 & HEAD（高频考点）

  • Branch

    • 只是一个指针

      📌补充：

      • 分支名本质是 .git/refs/heads/<branch> 里的一个 commit hash。

      • 所以创建分支几乎零成本。

    • 指向某个 commit

    • 默认分支：master / main

      📌补充：

      • 现在很多仓库默认是 main，但概念相同。

  • HEAD

    • 指向你当前所在的分支

      📌补充：

      • 通常 HEAD 里存的是 ref: refs/heads/master 这种“引用”。

      • 如果你 checkout <hash>，HEAD 会变成直接指向某个 commit（分离 HEAD）。

    • HEAD → branch → commit

      📌补充（考点句式）：

      • 提交发生时：当前分支指针会往前移动到新 commit；HEAD 跟着它。

      • 所以“当前分支”其实就是“HEAD 指向的分支”。

👉 分支 ≠ 复制一份代码
👉 分支 = 指针（非常轻量）

📌补充：

四、Git 是分布式的（Distributed）

  • 每个 clone：

    • 都是一个完整仓库

      📌补充：

      • “完整”= 带全历史、全对象。

      • 所以离线也能看 log/diff。

  • 每个人都能：

    • 独立提交

    • 独立分支

  • 再通过 push / pull 同步

📌补充：

  • 真正需要网络的是“交换对象”：fetch/push。

  • pull 只是“拉下来并合并/变基”的快捷组合。

📌 好处：

  • 不怕服务器挂

  • 本地就是备份

📌补充：

  • 注意：本地是“版本历史备份”，但不是完整灾难备份（因为你电脑也会坏）。

  • 所以你后面备份章节才重要：Git 不能替代 3-2-1。

五、文件状态模型（非常重要）

1️⃣ 四种状态

状态 含义
Untracked Git 不知道的文件
Unmodified 和上次 commit 一样
Modified 改了但没暂存
Staged 已加入下次 commit

📌补充（把表变成“动作模型”）：

  • Untracked → git add → Staged

  • Modified → git add → Staged

  • Staged → git commit → Unmodified

  • Unmodified → 编辑文件 → Modified

2️⃣ Tracked vs Untracked

  • Tracked

    • 在 Git 管理之下

    • 可能是：

      • unmodified

      • modified

      • staged

📌补充：

  • “tracked”并不等于“没改”，而是“Git 已经认识它”。

  • 一旦某文件被 add 并提交过，它就成 tracked（除非你删除并提交删除）。

  • Untracked

    • 新文件

    • 常通过 .gitignore 忽略

📌补充：

  • .gitignore 只对“未被跟踪的文件”有效。

  • 如果一个文件已经 tracked，你再写进 .gitignore，它仍然会被追踪；需要 git rm --cached <file> 才能停止追踪（保留工作区文件）。

3️⃣ 常用命令

👉 一切状态问题，先 git status

📌补充（git status 的输出你要会读）：

  • “Changes not staged for commit” = Modified（还没 add）

  • “Changes to be committed” = Staged（已经 add）

  • “Untracked files” = Untracked

  • 同时它会提示你下一步可用命令（考试非常友好）。

七、一次完整的 Git 操作流程（Walkthrough）

典型流程：

📌逐句补充解释（每一行的作用）：

  • git init

    • 在当前目录创建仓库，让 Git 开始管理。

  • echo "version 1" > test.txt

    • 创建/覆盖写入文件内容。

    • 这里模拟“初始版本”。

  • git add test.txt

    • 把 test.txt 放入暂存区（Staged），表示“准备把它纳入下一次快照”。

  • git commit -m "first commit"

    • 生成第一个 commit，记录当下快照。

    • 考试点：commit 是 DAG 节点。

    • 修改已跟踪文件 → 状态变为 Modified。

  • echo "new file" > new.txt

    • 创建新文件 → 状态是 Untracked。

  • git add test.txt new.txt

    • 把已修改文件和新文件都加入暂存区。

    • 注意：add 不是“提交”，只是选中要提交的内容。

  • git commit -m "second commit"

    • 第二次快照：包括 test.txt 新内容 + new.txt。

八、分支（Branching）——Git 的灵魂

1️⃣ 创建分支

📌逐句补充解释：

  • git branch feature

    • 创建一个名为 feature 的分支指针，指向当前 commit。

    • 不会切过去（你还在原分支）。

  • git checkout feature

    • 切换到 feature 分支：HEAD 指向 feature。

    • 工作区会更新为 feature 当前指向的快照。

  • git checkout -b feature

    • 等价于“创建 + 切换”一条龙。

    • 新手最常用。

  • 新分支指向当前 commit

  • HEAD 跳到新分支

📌补充：

  • 用 git status 你会看到 “On branch feature”。

2️⃣ 为什么分支这么牛？

  • 创建几乎零成本

  • 鼓励：

    • 小步提交

    • 实验性开发

  • 比 SVN 强太多

📌补充：

  • 团队中常见约定：

    • main/master：稳定

    • feature/*：新功能

    • fix/*：修 bug

  • 对应协作流程：PR/MR。

九、合并（Merge）

1️⃣ 合并流程

📌逐句补充解释（每条指令作用 + 场景）：

  • git checkout master

    • 先切到接收方分支（你要把别人合进来的那个分支）。

    • 这里 master 是“主线”。

  • git merge iss53

    • 把 iss53 分支合并到当前分支 master（核心规则：merge 的结果落在当前分支）。

    • 可能出现两种结果：

      1. fast-forward：master 没改过，只是落后 → 指针直接前移，无 merge commit

      2. merge commit：两边都改过 → 产生新的 merge commit（两个父节点）

  • git branch -d iss53

    • 删除本地分支指针 iss53（只删“指针”，不删提交对象）

    • -d 是安全删除：只有在 iss53 的提交都已合并时才允许删除。

    • 强制删除用 -D（危险，可能丢失你未合并的工作）

  • 创建 merge commit

  • 有两个父节点

📌补充：

  • 只有在“非 fast-forward”时才会真正创建 merge commit。

  • 考试写法可以说：merge 可能产生 merge commit。

2️⃣ 冲突（Conflict）

发生条件：

  • 两个分支

  • 修改了同一个文件的同一位置

📌补充：

  • 同一文件不同位置通常能自动合并。

  • 二进制文件（如图片）通常无法自动合并，常直接要求你选版本。

Git 会插入：

📌补充：

  • HEAD 表示你当前分支那边的内容。

  • commit-hash 表示被合并进来的那边对应的提交。

解决步骤：

  1. 手动改文件

  2. git add

  3. git commit

📌补充：

  • git add 在这里的含义是：“标记冲突已解决，把最终版本加入暂存区”。

  • commit 生成一个 merge commit（或普通 commit，取决于情况），完成合并过程。

十、Rebase（高阶，但常考）

1️⃣ Rebase 的本质

把一个分支的提交
“搬到”另一个分支的最前面

📌补充：

  • “搬”不是物理移动，是复制提交并改 parent，所以 commit hash 会变。

  • 结果：历史看起来像“一条直线”。

  • 不产生 merge commit

  • 历史是线性的

📌补充：

  • 线性历史好读，但代价是可能“重写历史”。

2️⃣ 示例

📌逐句补充解释：

  • git checkout experiment

    • 切到要被重排历史的分支（实验分支）。

  • git rebase master

    • 把 experiment 上“独有的提交”复制一份，接到 master 最新提交之后。

    • 复制完成后，experiment 指向新复制出来的提交链。

  • git checkout master

    • 切回主分支。

  • git merge experiment

    • 此时 master 只是落后，通常是 fast-forward 合并 → 不会产生 merge commit。

    • 这就是常见的“rebase + ff merge”流程。

📌 记忆点：

Merge Rebase
保留历史 重写历史
有 merge commit 无 merge commit（常见）
更安全 更干净

📌补充：

  • “更安全”指 merge 不会改旧提交的 hash，协作中不容易把别人搞崩。

  • “更干净”指 rebase 历史线性，但要遵守公共分支禁忌。

⚠️ 永远不要 rebase 已经 push 的公共分支

📌补充（为什么）：

  • 因为你 rebase 后提交 hash 全变了，远程和别人本地历史对不上，会逼大家做强制同步（很容易把别人工作覆盖）。

📌逐条补充解释：

  • git remote -v

    • 列出远程仓库的 fetch/push URL。

    • 场景：你怀疑 push 到了错误仓库、或者要检查是 SSH 还是 HTTPS。

  • git remote show origin

    • 展示 origin 的详细信息：有哪些远程分支、哪些本地分支在跟踪它。

    • 场景：排查跟踪关系（upstream）、分支是否落后。

  • git fetch origin

    • 只下载远程的新对象 + 更新本地的 origin/* 指针

    • 不改工作区、不动你当前分支指针。

⚠️ git fetch：

  • 只更新本地数据库

  • 不改工作区

📌补充：

  • fetch 之后你通常会：

    • git log --oneline --graph --decorate --all 看远程发生了什么

    • 再决定 merge 或 rebase 或 reset（看团队规范）。

十二、Pull & Push

1️⃣ Pull = Fetch + Merge

📌补充：

  • 默认 pull 是 fetch + merge，但也可能被配置成 fetch + rebase：

    • git pull --rebase（线性历史团队常用）

  • 场景：你想“一键同步并把远程改动合进来”。

  • 风险：pull 会直接改变你的当前分支/工作区；如果你本地有未提交修改，可能产生冲突或失败。

2️⃣ Push

📌补充：

  • 作用：把本地 branch 的提交上传到 origin，更新远程分支指针。

  • 常见简写：在已设置 upstream 后可用 git push。

  • 典型场景：

    • 提交作业到远程

    • 发 PR 前 push feature 分支

注意：

  • 本地分支 ≠ 自动同步

  • 可以有私有分支

📌补充：

  • 你可以在本地创建很多实验分支，不 push 也没关系。

  • push 的本质是“共享”，不是“必须”。

📌 好习惯：

push 前先 pull

📌补充：

  • 更严格说：先 fetch 看一下，再 merge/rebase 再 push，更安全：

十三、标准 Git 工作流（必背）

📌逐句补充解释（让你“背得有意义”）：

  • git checkout master：先从主线分支出发，确保 feature 基于最新主线。

  • git checkout -b feature：创建并切换到功能分支，隔离开发。

  • git add .：把本次要提交的修改加入暂存区（. 代表当前目录下所有变更，考试/实战都要会用，但也要知道它可能把不该提交的文件加进来）。

  • git commit -m "xxx"：形成一个清晰的“逻辑提交单元”。

  • git push origin feature：把 feature 分支发到远程，便于备份/协作/开 PR。

  • git checkout master：合并前回到主线。

  • git pull origin master：先同步主线（避免你合并时基于旧的 master）。

  • git merge feature：把 feature 合并进 master（可能 ff 或 merge commit）。

  • git push origin master：把合并结果推上远程，主线更新完成。

📌补充：

  • 很多团队现在用 PR/MR 来做 merge feature -> master，但本质一样。

💾 Backups（备份）——血泪教训

一、一句话总结

Your shit will break.

📌补充：

  • 这句的“考试含义”：系统设计必须假设失败是常态，而不是意外。

  • 备份/容灾属于“工程现实主义”。

二、为什么一定要备份

Murphy 定律：

  • 误删 / 恶意删除

  • 硬件损坏

  • 软件 Bug

  • 被偷（Berkeley 特供）

📌补充：

  • Git 解决不了：硬盘坏 / 电脑丢 / 云端账号被封

  • 所以备份要“多份 + 异地 + 可恢复”。

三、备份基本原则

  1. 自动化

    📌补充：

    • 手动备份=迟早忘。

    • 常见方式：cron 定时、CI 定时、备份软件计划任务。

  2. 安全（防泄露）

    📌补充：

    • 备份里往往包含密钥、隐私数据。

    • 需要：加密、权限控制、最小可见范围。

  3. 定期测试恢复

    📌补充：

    • 备份的唯一价值=能恢复。

    • 常见灾难：备份文件存在但损坏、备份脚本早就失败没人发现。

👉 不能只“有备份”，还要“能恢复”

📌补充：

  • 这句话在面试/考试里等于：备份必须包含恢复演练。

四、3-2-1 备份法则（考试必考）

  • 3 份数据

  • 2 种介质

  • 1 份异地

📌补充（为什么这样设计）：

  • 3 份：至少 1 份坏了还有替代

  • 2 种介质：避免同类介质系统性故障（同批硬盘、同一 NAS）

  • 1 异地：防火灾/盗窃/地震/勒索病毒（同地点同时完蛋）

例子：

  • 本地硬盘

  • 另一台电脑

  • 云端 / 异地

📌补充：

  • “异地”不一定是云，也可以是另一栋楼、另一城市机房。

五、GitLab 事故（经典案例）

事故过程

  • 工程师误 rm -rf

    📌补充：

    • 典型人为事故：一个命令把重要目录清空。

    • 说明：再强的系统也挡不住“误操作 + 无演练”。

  • 300GB 数据丢失

  • 备份：

    • S3 失败（几周没成功）

      📌补充：

      • 备份策略存在 ≠ 备份成功。

      • 没监控就等于没备份。

    • Azure 快照没开

      📌补充：

      • “没开”=压根没配置。很多事故都不是技术做不到，而是流程没做。

    • 最后靠 LVM 勉强救回

      📌补充：

      • 这是运气救场，不可复制，不能当策略。

结果

  • 丢失：

    • 5000 项目

    • 5000 评论

    • ~700 用户

📌补充：

  • 这些损失不仅是代码，还有元数据（issue、评论、用户关系），所以仅靠 Git 仓库备份不够。

📌 教训：

  • 备份要 监控

  • 要 演练恢复

  • 不测试 = 没备份

📌补充：

  • 监控：备份作业成功/失败告警

  • 演练：定期在新机器上恢复一次，验证“端到端可用”。

六、常见备份工具

1️⃣ rsync

📌逐项补充（每个参数作用）：

  • rsync：远程同步工具（本地↔远程 / 本地↔本地都行）

  • -a（archive）：递归 + 保留权限/时间戳/软链接等（常用的“完整同步模式”）

  • -v（verbose）：输出更详细信息（便于看在干啥）

  • -P：相当于 --progress + --partial

    • --progress 显示进度

    • --partial 断点续传（中断后不必从头传）

  • source：源路径（文件/目录）

  • user@host:dest：目标（通过 SSH 到远程主机的路径）

  • 增量

  • 走 SSH

  • 本地 ↔ 远程

📌补充（“增量”是什么意思）：

  • rsync 会比较文件大小/时间戳/校验信息，只传变化部分，大幅节省流量。

📌应用场景：

  • 每天夜里把项目目录同步到另一台机器

  • 同步网站静态资源到服务器

  • 结合 cron：自动化备份

⚠️常见坑（很重要）：

  • rsync 末尾有没有 / 含义不同：

    • rsync -a src/ dest/：把 src 目录内容同步到 dest

    • rsync -a src dest/：把整个 src 目录作为子目录放进 dest

2️⃣ rclone

📌逐项补充：

  • rclone：面向云存储的同步工具（“云端 rsync”）

  • sync：让目标端变成和源端一致（目标端多余的文件会被删除）

  • source:path / dest:path：rclone 配置里的“远程端”或本地路径

  • 云端版 rsync

  • 支持：

    • Google Drive

    • S3

    • OneDrive

  • 可挂载为本地文件系统

📌应用场景：

  • 实现 3-2-1 的“异地 1 份”

  • 把关键资料同步到云端（并加密/权限控制）

⚠️重要提示：

  • 因为 sync 会删除目标端多余文件，所以目标端别拿来存别的东西，否则可能被清掉。

  • 更保守的命令是 copy（只复制不删除）。
